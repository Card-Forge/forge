     Deck Piloting Guidance for the Forge LLM Sidecar

     Context

     The forge-llm-sidecar today runs a single LangGraph node — deck recognition
     — which guesses the opponent's archetype and writes it to the game log. It
     does not advise the AI on how to play its own deck.

     This change adds a piloting guidance layer: per-archetype JSON knowledge
     files (mulligan rules, game plan, win conditions, key cards, matchup notes,
     meta threats) validated by Pydantic, used by the LLM to recommend what the
     AI
     should play next.

     Because hand-authoring ~60 guides per format is infeasible, an offline
     LLM-assisted extraction script generates schema-valid guides from
     MTGGoldfish
     decklists/primers — mirroring the existing scrape_metagame.py pattern. The
     runtime sidecar stays fully offline.

     Design decisions (confirmed with the user)

     - One node, one LLM call. Deck recognition and piloting advice are merged
     into a single game_advisor node. The AI re-runs the graph on every action,
     and a second LLM call would add noticeable lag — so a single prompt returns
     both the opponent guess and the play recommendation. This deliberately
     overrides the "one concern per node" convention in docs/EXTENDING.md;
     latency is the priority.
     - Own archetype is identified once per game, no LLM. On the first
     /recognize call for a game_id, the sidecar matches the AI's deck_cards
     (already sent today) against known archetypes by signature-card overlap and
     caches the result per game_id — reusing the existing _resolved_format
     cache pattern. No Forge-side change.
     - Extraction: offline LLM-assisted script; builder LLM is configurable,
     defaulting to the existing local llama.cpp client.
     - Forge's Java adapter changes that capture live hand/board state are a
     larger, separate effort — this plan ships the sidecar end-to-end and
     updates
     the Java request/result records + docs, but defers the state-capture
     wiring.

     Knowledge layer

     1. Pydantic schema — app/knowledge/piloting_schema.py (new)

     PILOTING_SCHEMA_VERSION = 1. Models:
     - StrategyType(str, Enum) — aggro | tempo | midrange | control | combo | 
     ramp
     - MulliganHandExample — hand: list[str], decision: 
     Literal["keep","mulligan"], reason: str
     - MulliganGuidance — keep_criteria: list[str], mulligan_criteria: 
     list[str], examples: list[MulliganHandExample]
     - GamePlan — early_game / mid_game / late_game: list[str]
     - KeyCard — name: str, role: str, notes: str = ""
     - MatchupNote — opponent_archetype: str, advice: str, watch_for: list[str]
     - GuideMetadata — source, generated_at, model, schema_version
     - PilotingGuide — archetype, format, strategy_type: StrategyType,
     overview, win_conditions: list[str], mulligan: MulliganGuidance,
     game_plan: GamePlan, key_cards: list[KeyCard], sequencing_tips: list[str],
     matchups: list[MatchupNote], common_threats: list[str], metadata.

     2. Guide files — app/knowledge/piloting/ (new)

     - piloting/<format>/<slug>.json — archetype-specific (e.g.
     modern/boros-energy.json).
     Slug = lowercased alphanumerics joined by hyphens (helper alongside
     loader._norm).
     - piloting/generic/{aggro,tempo,midrange,control,combo,ramp}.json — six
     hand-authored, committed fallbacks.
     - Initial archetype guides: generate the Modern set via the builder script
     and commit after human review; other formats follow later.

     3. Loader — app/knowledge/piloting.py (new)

     - get_piloting_guide(archetype, game_format, strategy_type=None) -> 
     PilotingGuide | None
     — functools.cache per-file load; parse JSON, validate with PilotingGuide;
     on pydantic.ValidationError log a warning and skip (fail-soft, like
     loader._load_file / metagame._load). Fallback chain:
     format/<slug> → generic/<strategy_type> → generic/midrange.
     - identify_own_archetype(deck_cards, game_format) -> tuple[str|None, 
     StrategyType]
     — deterministic, no LLM: score each metagame / curated archetype by
     signature-card overlap with deck_cards; return the best match (name +
     strategy type). Used once per game (cached by the node, see §5).
     - available_guides() — debug helper.

     Extend curated app/knowledge/archetypes/*.json entries with an optional
     strategy_type field so the generic fallback can be chosen; default midrange
     when absent. merge_with_curated carries it through unchanged.

     Offline extraction — scripts/build_piloting_guides.py (new)

     CLI mirroring scripts/scrape_metagame.py:
     python scripts/build_piloting_guides.py                 # all formats
     python scripts/build_piloting_guides.py modern          # one format
     python scripts/build_piloting_guides.py modern --archetype "Boros Energy"
     python scripts/build_piloting_guides.py modern --force  # regenerate
     existing
     Per archetype: read metagame_data/<format>.json → fetch a representative
     decklist (and primer text when present) via new helpers in
     app/knowledge/scraper.py (fetch_archetype_decklist / fetch_primer_text)
     → build prompt → builder LLM → parse JSON → validate PilotingGuide →
     write piloting/<format>/<slug>.json. Skips existing files unless --force;
     polite time.sleep; succeeds if ≥1 guide written. Generic guides are NOT
     generated here (hand-authored).

     Builder LLM — app/knowledge/builder_llm.py (new): a thin wrapper over the
     same OpenAI-compatible chat API as llm_client.py, configured by
     BUILDER_LLM_BASE_URL / BUILDER_LLM_API_KEY / BUILDER_MODEL_NAME, each
     falling back to the existing LLM_* values in app/config.py. This lets a
     stronger cloud model generate guides offline while runtime stays local.

     Runtime: merged game_advisor node

     4. Schema — app/schema.py

     - RecognitionRequest: add optional fields (defaults keep old clients
     valid):
     hand: list[str], own_board: list[str], opponent_board: list[str],
     your_graveyard: list[str], opponent_graveyard: list[str],
     life_totals: dict[str,int].
     - RecognitionResponse: keep the existing recognition fields; add a
     piloting: PilotingAdvice | None sub-model (recommended_play, reasoning,

     SCHEMA_VERSION to 2.
     - GraphState: add input keys (hand/board/graveyards) and output keys
     (own_archetype, piloting_guide, recommended_play, play_reasoning,
     play_alternatives, mulligan_advice) on top of the existing recognition
     keys.

     5. Node — rename deck_recognition → game_advisor

     Rename app/nodes/deck_recognition.py → app/nodes/game_advisor.py and evolve
     its single node:
     1. Own archetype — _own_archetype: dict[str,tuple[str,StrategyType]] cache
     keyed by game_id (mirrors _resolved_format). First call:
     identify_own_archetype(deck_cards, format); later calls hit the cache.
     2. Resolve metagame slug + candidate opponent archetypes (unchanged logic).
     3. Load the AI's own PilotingGuide via the fallback chain.
     4. One prompt, one generate_json call asking for a single JSON object with

     cards
     / matchup note for the suspected opponent), and the live
     hand/board/graveyard
     state.
     5. Coerce/clamp every field as deck_recognition does today; fail-soft on
     LLMError — return neutral recognition and neutral piloting values.
     Always returns a valid GraphState (per docs/EXTENDING.md).

     6. Graph — app/graph.py

     START → game_advisor → END (single node; constant renamed).

     Research note (extraction source)

     No site exposes structured piloting data via an API. MTGJSON/Scryfall cover
     cards only; primers (MTGGoldfish articles, Mystical Teachings, Draftsim)
     are
     free prose with patchy archetype coverage. Hence the chosen approach: feed
     a
     representative decklist (always available from MTGGoldfish archetype pages)
      plus
     any primer prose to an LLM and have it emit schema-valid JSON — the only
     option
     that scales to every metagame archetype. Generated files are committed and
     human-reviewable, so quality stays inspectable.

     Verification

     1. cd forge-llm-sidecar && pip install -e ".[dev]"
     2. ruff check . && black --check . && pytest — all green (LLM stubbed
     offline).
     3. Schema/loader: python -c "from app.knowledge.piloting import 
     get_piloting_guide; print(get_piloting_guide('Boros Energy','modern'))".
     4. Builder (needs an LLM running):
     python scripts/build_piloting_guides.py modern --archetype "Boros Energy"
     → writes a valid piloting/modern/boros-energy.json.
     5. Start the sidecar; GET /piloting?format=modern&archetype=Boros%20Energy
     returns the guide.
     6. POST /recognize with a synthetic body including hand / own_board /
     deck_cards → one response carries both an opponent archetype guess and
     a piloting block (recommended_play, own_archetype); a second call for
     the same game_id reuses the cached own archetype. Omitting the new fields
     still returns a valid response (back-compat).
     7. Stop the LLM server and repeat (6): the single node degrades fail-soft
     (Unknown guess, neutral piloting advice) — the request still succeeds.
