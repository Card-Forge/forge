     Self-Play Results Store: baselining + over-time tracking

     Context

     Self-play results today are ephemeral and disconnected from time.
     SelfPlayRunner
     writes per-seat JSONL to a user-named, gitignored file
     (selfplay/runs/*.jsonl), and
     scripts/selfplay_report.py aggregates whatever files you point it at into a
      win-rate
     table at query time. Nothing is persisted with a timestamp, nothing records
      a deck's
     baseline, and — critically — nothing links a run's results to the state of 
     the
     learnings store when that run happened.

     That last gap is the whole point of "see how the self-improvement system
     works over time":
     the sidecar already computes a learnings_version() cache-bust token
     (app/knowledge/learnings.py) that changes whenever lessons are staged. If
     every run
     captures that token alongside its win rate, you can answer "did win-rate / 
     turns-to-win
     move after the learnings changed?" — the self-improvement signal.

     Decisions (confirmed with user): SQLite (stdlib, single local file,
     offline-first —
     not Supabase); ingest via a script (Java runner stays unchanged, keeps
     emitting JSONL as
     the raw artifact); surface via a CLI trend report and a dashboard panel.

     Approach

     Keep the existing pipeline as-is (runner → JSONL → selfplay_report.py for
     one-shot
     file aggregation). Add a persistent layer on top:

     SelfPlayRunner → runs/*.jsonl  (raw artifact, unchanged)
                           │  scripts/record_run.py  (ingest + metadata capture)
                           ▼
                   selfplay/results.db  (SQLite)
                      ├── scripts/selfplay_trends.py     (CLI: baseline vs
     latest, trend)
                      └── GET /api/selfplay/trends → dashboard "Self-play
     trends" panel

     1. SQLite store module — app/selfplay_store.py (new)

     Importable by both the ingestion script and the sidecar endpoint. Uses
     stdlib sqlite3
     (no new dependency). DB path defaults to selfplay/results.db, overridable
     via env
     FORGE_SELFPLAY_DB (mirrors the config-via-env pattern in app/config.py).

     Two tables — a run = one ingested JSONL batch (one runner invocation); a
     game =
     one seat-record:

     CREATE TABLE runs (
       run_id INTEGER PRIMARY KEY AUTOINCREMENT,
       created_at TEXT NOT NULL,        -- ISO8601 ingestion time
       config TEXT,                     -- "goldfish" | "mirror"
       format TEXT,                     -- "standard", "modern", ... (from 
     --format flag)
       learnings_version TEXT,          -- learnings.learnings_version() at 
     ingest time
       git_sha TEXT,                    -- `git rev-parse --short HEAD`, 
     best-effort
       source_file TEXT,                -- ingested JSONL path
       label TEXT                       -- optional note / "baseline" pin
     );
     CREATE TABLE games (
       id INTEGER PRIMARY KEY AUTOINCREMENT,
       run_id INTEGER NOT NULL REFERENCES runs(run_id),
       archetype TEXT, opponent TEXT, pilot_mode TEXT,
       won INTEGER, win_turn INTEGER, turns INTEGER
     );
     CREATE INDEX idx_games_run ON games(run_id);

     Functions:
     - connect(path=None) — open DB, run CREATE TABLE IF NOT EXISTS (idempotent
     migration).
     - insert_run(meta, records) — write one run row + its game rows in a
     transaction; returns run_id.
     - archetype_trend(archetype=None) — per (archetype, opponent, pilot_mode),
     return the
     time-ordered series of runs with created_at, learnings_version, and
     aggregates
     (n_games, n_wins, win_rate, fastest_win, mean_win_turn, median_win_turn).
     Aggregates
     computed in SQL so the store is the single source of truth.
     - baseline_vs_latest() — per group, the baseline (run flagged
     label='baseline',
     else the earliest run) and the latest run, with deltas (win_rate,
     mean_win_turn).
     This is the headline baselining view.

     Reuse the existing turn-stat math shape from selfplay_report._summarize
     (win_turn only counted when won and numeric) so DB aggregates match the
     file report.

     2. Ingestion script — scripts/record_run.py (new)

     python -m scripts.record_run runs/ruby.jsonl --format modern --config
     goldfish [--label baseline]
     - Parse JSONL reusing selfplay_report._iter_records (avoid duplicate
     parsing).
     - Capture metadata the JSONL lacks: --format, --config, --label;
     auto-capture
     learnings_version() (import from app.knowledge.learnings) and git_sha
     (git rev-parse --short HEAD, swallow failure).
     - Call selfplay_store.insert_run(...); print the new run_id and a one-line
     summary.

     3. CLI trend report — scripts/selfplay_trends.py (new)

     python -m scripts.selfplay_trends                 # baseline-vs-latest 
     table, all decks
     python -m scripts.selfplay_trends --archetype "Ruby Storm"   # full time 
     series for one deck
     python -m scripts.selfplay_trends --json
     Renders, per deck: baseline win% → latest win%, Δwin%, Δmean-turn, #runs,
     and (for the
     single-archetype view) the per-run series with learnings_version so
     learnings changes
     line up against performance. Table-formatting mirrors selfplay_report.py's
     style.

     4. Sidecar endpoint — GET /api/selfplay/trends in app/main.py

     Sibling to /api/stats (around app/main.py:217). Returns
     selfplay_store.baseline_vs_latest() (and optional ?archetype= →
     archetype_trend) as
     JSON. Fail-soft: if the DB file doesn't exist yet, return {"groups": []}
     (empty panel,
     no error) — matches the dashboard's tolerant fetch pattern.

     5. Dashboard panel — app/static/dashboard.html

     Add a "Self-play trends" <section> and a fetchTrends() that polls
     /api/selfplay/trends, wired into the existing refresh() / setInterval loop
     (dashboard.html:1129, :1153). Render a per-deck table: deck, baseline →
     latest win%,
     Δ (green/red), mean turns trend, #runs. Pure DOM/table like the existing
     panels — no chart
     library. (A small inline sparkline of win% per run is a nice-to-have if
     cheap.)

● Done. Here's what I built — a SQLite-backed results store on top of your

  Why SQLite, not Supabase

  Supabase (hosted Postgres) would add a network dependency, auth, and hosting
  that directly contradict this repo's offline-first design — and it's overkill
  for single-user local self-play data. SQLite is Python stdlib (zero new deps),
   a single local file, and SQL makes the baseline/trend queries trivial.
  Confirmed with you before building.

  What changed

  New files:
  - app/selfplay_store.py — the store. Two tables: runs (one ingested JSONL
  batch + metadata) and games (one seat-record). Key functions: insert_run,
  archetype_trend, baseline_vs_latest. Aggregation math matches
  selfplay_report._summarize so numbers are consistent.
  - scripts/record_run.py — ingest a runner JSONL file, capturing what the JSONL
   lacks: timestamp, --format, --config, --label, the learnings_version() token,
   and git sha.
  - scripts/selfplay_trends.py — CLI: per-deck baseline-vs-latest table
  (default), or full per-run series with --archetype.
  - tests/test_selfplay_store.py — 6 tests (store round-trip, time ordering,
  deltas, baseline-pin override, endpoint empty + seeded).

  Edited: app/main.py (GET /api/selfplay/trends, fail-soft empty when no DB,
  excluded from hit-logging); app/static/dashboard.html ("Self-play Trends"
  panel wired into the 5s refresh loop); .gitignore (selfplay/*.db); AGENTS.md +
   README.md.

  Unchanged: SelfPlayRunner.java and selfplay_report.py (still your quick
  one-shot file aggregator).

  The core idea for "improvement over time"

  Each run snapshots learnings_version() — the token that changes whenever
  lessons are staged. So selfplay_trends.py --archetype X shows win% and
