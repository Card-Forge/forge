     Plan: Host the LLM sidecar so Forge works from anywhere

     Context

     Today the deck-recognition feature only works when Forge, the sidecar, and
     the
     llama.cpp LLM server all run on the same machine. The Forge Java client
     defaults
     to http://localhost:8000, and the sidecar's __main__ block binds to
     127.0.0.1 — both are localhost-only.

     The goal: run Forge on any device and still get the AI "thought process"
     (the
     in-game LLM deck guesses + the /dashboard recognition history). The chosen
     approach keeps the sidecar and llama.cpp on a home server, and makes them
     reachable from anywhere over a Tailscale tailnet — no public exposure, no
     cloud LLM cost, network-level auth/encryption handled by Tailscale
     (WireGuard).

     Because Tailscale provides the security boundary, no application-level API 
     key
     or HTTPS is needed — that keeps the change small. The only real work is
     making
     the sidecar bind beyond localhost, pointing the Forge client at the home
     server,
     and documenting the Tailscale setup.

     Architecture after this change

     [ laptop, anywhere ]                 [ home server ]
       Forge desktop  --Tailscale-->  sidecar :18970  --localhost-->  llama.cpp
     :8080
       browser /dashboard --Tailscale--^

     Forge connects to http://<home-server>.<tailnet>.ts.net:18970. The sidecar
     still calls llama.cpp at localhost:8080 because they share the host.

     Changes

     1. Let the sidecar bind beyond localhost — forge-llm-sidecar/app/config.py

     Add a host field to Config (default 127.0.0.1, env var HOST):

     host: str
     ...
     host=os.environ.get("HOST", "127.0.0.1"),

     2. Use it in the entrypoint — forge-llm-sidecar/app/main.py

     In the __main__ block, replace the hardcoded host="127.0.0.1" with
     host=CONFIG.host. Default stays localhost-only; running with HOST=0.0.0.0
     (set on the home server) exposes it to the tailnet.

     3. Forge launcher points at the sidecar — run-forge.sh

     Add, before the exec java line, env-var-driven system properties so the
     deck-recognition feature is enabled and aimed at the home server:

     # --- LLM sidecar (override host via FORGE_SIDECAR_URL) 
     -----------------------
     SIDECAR_URL="${FORGE_SIDECAR_URL:-http://localhost:18970}"
     SIDECAR_PROPS="-Dforge.ai.deckRecognition=true 
     -Dforge.ai.deckRecognition.url=$SIDECAR_URL"

     and add $SIDECAR_PROPS to the exec java argument list. On a remote machine
     the user runs FORGE_SIDECAR_URL=http://home-server.tailnet.ts.net:18970 
     ./run-forge.sh.
     This also fixes the existing port mismatch (Java client default 8000 vs
     sidecar 18970) and the feature being off by default.

     4. Run the sidecar as a service on the home server (new file)

     Add forge-llm-sidecar/scripts/forge-sidecar.service — a systemd unit
     template
     so the sidecar auto-starts and stays up (required for "from anywhere" to be
     reliable). It runs uvicorn app.main:app with HOST=0.0.0.0, PORT=18970, and
     LLM_BASE_URL=http://localhost:8080/v1. README documents systemctl --user
     install. (llama.cpp is assumed already managed by the user; the unit only
     covers the sidecar.)

     5. Documentation

     - forge-llm-sidecar/README.md / docs/PLAYING_WITH_SIDECAR.md: add a
     "Remote access over Tailscale" section — the Tailscale setup steps below,
     the HOST=0.0.0.0 requirement, the systemd unit, and the
     FORGE_SIDECAR_URL usage.
     - forge-llm-sidecar/AGENTS.md: note the new HOST config var.

     Tailscale setup (no code — operator steps, documented in README)

     1. Install Tailscale on the home server and on each device that runs Forge;
     tailscale up and sign into the same tailnet on all of them.
     2. Enable MagicDNS in the Tailscale admin console so the server has a
     stable name (e.g. home-server.tailnet-name.ts.net).
     3. Do not port-forward 18970 on the home router — Tailscale is the only
     ingress path.
     4. Optional hardening: a Tailscale ACL rule restricting :18970 to specific
     devices, and/or bind the sidecar to the host's Tailscale IP instead of
     0.0.0.0.

     What does NOT change

     - No API key / auth code, no CORS, no HTTPS — Tailscale is the security
     layer.
     - No Java client changes beyond the launcher; DeckRecognitionFeature
     already
     reads the forge.ai.deckRecognition.url system property.
     - The /dashboard, /recognize, /health routes are unchanged — the dashboard
     becomes reachable at http://<server>.<tailnet>.ts.net:18970/dashboard from
     any tailnet device's browser, which satisfies "AI thought process
     anywhere".
     - Docker path is left as-is (it already binds 0.0.0.0); running directly
     via
     the systemd unit is recommended so llama.cpp stays reachable at localhost.

     Verification

     1. Home server: start llama.cpp on :8080, then
     HOST=0.0.0.0 PORT=18970 uvicorn app.main:app (or install the systemd unit).
     curl http://localhost:18970/health →
     {"status":"ok","llm_reachable":true,...}.
     2. Tailnet reachability: from a second tailnet device,
     curl http://<server>.<tailnet>.ts.net:18970/health returns the same.
     3. Dashboard: open http://<server>.<tailnet>.ts.net:18970/dashboard in a
     browser on the remote device — page loads, /api/stats populates.
     4. Forge end-to-end: on the remote device,
     FORGE_SIDECAR_URL=http://<server>.<tailnet>.ts.net:18970 ./run-forge.sh,
     start an AI-vs-AI or vs-AI game, and confirm LLM deck guess: ... lines
     appear in the in-game log and new entries show on the dashboard.
     5. Fail-soft check: stop the sidecar; confirm Forge still plays normally
     (only a debug log line), proving the gate still degrades gracefully.

● Good questions — and the answer to the second one is reassuring. Here's how it
   actually works.

  How to verify a recognition request was made

  A /recognize call fires from the AI's observer whenever you (its opponent) 
  play a land, cast a spell, or a new turn begins (after turn 2). You can see it
   in four places, easiest first:

  1. Forge's in-game Game Log panel — the end result. Look for lines like:
  LLM deck guess: opponent is playing 'Izzet Spells' (confidence 72%)
  This is DeckRecognitionObserver.postGuess() writing to the game log. If you
  see these, the whole pipeline works.

  2. The sidecar's terminal (where python -m app.main runs) — one line per
  request:
  INFO:forge-llm-sidecar:recognize: client=forge game=... turn=3
  INFO:     127.0.0.1:xxxxx - "POST /recognize HTTP/1.1" 200 OK

  3. The dashboard — the "request count" badge increments and the history
  table/last-recognition panel updates. Or poll directly:
  curl -s localhost:18970/api/stats | python3 -m json.tool

  4. The Forge terminal — at game start it prints, once:
  DeckRecognition: enabled for AI player '...' via http://localhost:18970
  If you don't see that, the observer never attached (sidecar was down when the
  game started) — and no recognitions will ever happen that game. That's the
  thing to check first.

  Do you need to play slowly?

  No. The recognition is fully asynchronous and never blocks the game:

  - The call runs on a background thread (client.recognizeAsync(...)); Forge
  does not wait for it. The AI plays its turn with its normal instant heuristics
   regardless.
  - It uses latest-wins coalescing: if a recognition is already in flight when
  you act again, only one rerun is queued and fired against the newest game
  state when the current one finishes. Intermediate states are simply skipped.

  So playing fast just means you see fewer guesses (and they lag ~7s behind your
   action). Playing slowly means you see a guess for more intermediate states.
  Either way nothing breaks and the game never stalls.

  One important clarification

  The LLM call is observational/advisory only — the README is explicit: "it does
   not change how the heuristic AI plays." The /recognize response (archetype
  guess + piloting advice) is written to the game log for you to read; it does
  not drive the AI's actual decisions. So there's no "give the AI time to
  strategize" — its real plays come from the built-in heuristic engine and are
  instant. The sidecar is a running commentary, not the brain.

  If you want a persistent, easy-to-tail log instead of watching a terminal,
  install the scripts/forge-sidecar.service systemd unit from earlier — then

● Here's what's happening. The sidecar doesn't get the format from Forge — Forge
   only reports the generic game type "Constructed", never the specific format.
  So the sidecar tries to figure it out itself:

  1. resolve_meta_format("Constructed") → None (ambiguous).
  2. infer_format(...) — looks up card legalities on Scryfall and picks the
  narrowest format. Important: it uses the AI opponent's decklist (the one Forge
   knows exactly), not yours.
  3. If that's disabled or fails → falls back to DEFAULT_META_FORMAT, which
  defaults to standard.

  So you get "standard" if either the AI opponent's deck happens to be
  Standard-legal (narrowest match wins), or Scryfall detection fails.

  How to force the format

  Set these two environment variables on the sidecar and restart it:

  FORMAT_DETECT_ENABLE=false DEFAULT_META_FORMAT=modern python -m app.main

  - FORMAT_DETECT_ENABLE=false turns off the Scryfall guessing.
  - DEFAULT_META_FORMAT=modern makes it always use Modern.

  Valid values (these have metagame data): standard, pioneer, modern, legacy,
  vintage, pauper, commander.

  If you installed the systemd unit, add them there instead:
  Environment=FORMAT_DETECT_ENABLE=false
  Environment=DEFAULT_META_FORMAT=modern

  If you'd rather keep auto-detection but just fix the fallback for when it
  fails, set only DEFAULT_META_FORMAT=modern and leave FORMAT_DETECT_ENABLE
  alone.

  Verify it worked

  After restarting, watch the sidecar log during a game — you'll see:
  game_advisor: game <id> -> metagame 'modern'

  The deeper fix (optional)

  The real limitation is that Forge sends "Constructed" instead of the actual
  format. Forge does know the match's format restriction — the proper fix is to
  have the Java side send it in the RecognitionRequest (format field) so the
  sidecar doesn't have to guess at all. That's a code change to
  DeckRecognitionObserver/RecognitionRequest plus reading the format from
  Forge's game rules. Want me to implement that? It'd make the format correct
