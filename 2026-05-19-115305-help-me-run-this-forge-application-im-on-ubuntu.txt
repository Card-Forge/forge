● Done. Forge now launches and the gauntlet screen no longer crashes. Here's
  what was wrong and what I changed.

  Why ./forge.sh did nothing

  Three separate problems, and the first one hid the others:

  1. No display. You're in tmux, which doesn't inherit $DISPLAY. The Swing GUI
  had nowhere to draw.
  2. Wrong working directory. Stock forge.sh does cd into
  forge-gui-desktop/target/, which has no res/ folder. Forge crashed with Cannot
   find default skin .../res/skins/default/bg_splash.png. The res/ resources
  actually live in forge-gui/.
  3. Silent failure. Forge's own error handler (BugReporter) failed its static
  initializer when the crash happened, so the JVM printed only
  ExceptionInInitializerError thrown from the UncaughtExceptionHandler — no
  useful detail. That's why "nothing happened."

  Why the gauntlet screen crashed

  Not your extension code. Gauntlet .dat files are XStream-serialized and
  contain a java.util.EnumMap. On Java 17, XStream needs --add-opens 
  java.base/java.util=ALL-UNNAMED to deserialize that. Without it you get
  exactly the ConversionException / Map$Entry.getValue() NPE you pasted. The
  assembled jar's manifest intends to supply this, but it's lost the moment
  anything launches via classpath instead of -jar.

  What I changed (you chose "Both")

  1. New launcher: run-forge.sh (repo root) — run Forge with this from now on:
  ./run-forge.sh
  It auto-detects DISPLAY (found :1), sets XAUTHORITY to GDM's cookie, cds into
  forge-gui/ so res/ is found, and passes the full set of --add-opens flags
  explicitly so it works whether launched as a jar, from Maven, or from an IDE.

  2. Graceful gauntlet loading — forge-gui/.../gauntlet/GauntletIO.java: a
  gauntlet file that fails to deserialize now logs one line and is skipped,
  instead of popping a modal bug-report dialog. Safety net if a file is ever
  genuinely incompatible. (Also removed the now-unused BugReporter import.)

  3. Rebuilt the jar with mvn -pl forge-gui,forge-gui-desktop -am -DskipTests -o
   package.

  Verified

  Ran ./run-forge.sh — Forge launched, GUI opened, stayed alive, and forge.log
  shows zero gauntlet/ConversionException/EnumMap errors. The gauntlet contests

● Fixed. The crash was a thread-safety bug, and it's pre-existing — not from the
   merge (I checked: the only upstream commit touching these files was the
  minlog→tinylog migration).

  Root cause

  Downloading net decks runs on a background thread (GuiDownloadZipService). As
  it downloads and extracts the zip, it calls reset(), setMaximum(), setValue(),
   setDescription() directly on FProgressBar — a Swing JProgressBar — from that
  background thread. Meanwhile the EDT is painting the same bar. Mutating a
  Swing component off the EDT during paint corrupts BasicProgressBarUI's
  internal cache, so componentInnards is seen as null mid-paint → the NPE you
  got. It's timing-dependent, which is why it didn't always show up.

  The base GuiDownloadService carefully wraps its updates in invokeInEdtLater,
  but GuiDownloadZipService (the net-deck path) never did.

  The fix

  Made FProgressBar itself thread-safe — the right place, since it implements
  IProgressBar and is explicitly driven by background download services. In
  forge-gui-desktop/.../toolbox/FProgressBar.java:

  - Added a runOnEdt() helper — runs immediately if already on the EDT, else
  invokeLater.
  - setValue(), setDescription(), reset(), and a new setMaximum() override now
  funnel through it, so any off-EDT caller is automatically marshaled.
  invokeLater preserves call order, so progress still advances correctly.
  - Marked the display-state fields (desc, percentMode, showETA, showCount)
  volatile for cross-thread visibility.

  This fixes the bug for every download flow (net decks, card images, skins,
  etc.), not just this one.
