  The latency question (this should drive your sizing decisions)

  LLM latency splits into two phases, and they scale very differently:

  - Prefill — processing the input prompt. Scales ~linearly with input tokens.
  Parallelizable, so relatively cheap per token.
  - Decode — generating output tokens one at a time (autoregressive). Each token
   is a full forward pass, so it's the usual bottleneck — but only matters in
  proportion to output length.

  Your sidecar emits small JSON (a few hundred output tokens), so for you
  input/prefill dominates. That's exactly the lever your guidance-doc sizing
  controls.

  Rough throughput on a local llama.cpp setup (your default localhost:8080):

  ┌────────────┬──────────────────────────────────┬──────────────────────────┐
  │            │ Decent GPU (~3000 tok/s prefill) │ CPU (~150 tok/s prefill) │
  ├────────────┼──────────────────────────────────┼──────────────────────────┤
  │ 2,000 in   │ ~0.7s                            │ ~13s                     │
  ├────────────┼──────────────────────────────────┼──────────────────────────┤
  │ 10,000 in  │ ~3.3s                            │ ~67s                     │
  ├────────────┼──────────────────────────────────┼──────────────────────────┤
  │ 100,000 in │ ~33s                             │ ~11 min                  │
  └────────────┴──────────────────────────────────┴──────────────────────────┘

  So is the difference meaningful? Yes, very — and it's roughly linear. 2k→10k
  is a 5x prefill cost. 100k is a non-starter for per-turn calls: it needs a
  128k-context model, tens of GB of KV-cache memory, and quality degrades in the
   middle of long contexts ("lost in the middle").

  The crucial caveat: KV prefix caching. If a chunk of the prompt is static and 
  identical across calls (like a piloting guide), llama.cpp can cache its
  prefill and reuse it — so you pay that cost once, not every call. Your sidecar
   already does whole-response LRU caching; the win is to also structure prompts
   so the static guidance is a stable prefix.

  Recommended budget:
  - Per-call dynamic content (game state, observations): keep < ~2-4k tokens →
  snappy.
  - Static guidance injected per archetype: your guides currently format to
  ~2,900-3,370 tokens. That's fine if cached as a prefix; otherwise trim toward
  ~1,500-2,500.
  - Goldfishing-learned notes appended to a guide: distill hard — ≤ ~500 tokens 
  of high-signal tips, never raw logs. The whole point is compression: "Turn-3
  kill needs 2 rituals + reducer by T2" not a transcript.

  The trap to avoid: dumping accumulated goldfishing data straight into prompts.
   That's how you'd drift toward 100k. The system must distill before it writes
  back.

  Goldfishing system — the shape of it

  You already have the hard parts. SimulateMatch +
  GameSimulator/GameCopier/SpellAbilityPicker give you headless games and
  lookahead. Goldfishing = run a deck vs. a passive opponent, measure
  turns-to-win, repeat thousands of times, distill what correlates with fast
  kills, write it back to the piloting JSON.

  The big fork is who plays the deck during goldfishing, because it changes
  everything downstream:

  - A) Built-in Forge AI — fast, deterministic, no LLM, scales to thousands of
  games trivially. Data reflects what Forge's AI does, which may be suboptimal
  goldfishing.
  - B) Search for the optimal/fastest line — use the existing GameSimulator
  lookahead (beam/MCTS over action sequences) to actually find the fewest-turn
  kill. This is "true" goldfishing and produces the most valuable guidance, but
  it's more to build.
  - C) The LLM sidecar itself — slow, but tests/improves the actual system.
  Realistically only viable at low volume.

  Target: pros/cons + my recommendation

  Option A — write distilled tips straight into the curated piloting JSON guides
  - Pros: single source of truth; _format_guide() (game_advisor.py:116) already
  injects it, zero new plumbing.
  - Cons: machine noise pollutes hand-curated guides; no provenance (can't tell
  learned vs. authored); hard to roll back; guides grow unbounded → token bloat
  → slower prefill (the exact thing your latency budget is fighting).

  Option B — separate learnings layer, injected alongside guides
  - Pros: clean provenance; you can cap injected size, gate by
  confidence/sample-count, A/B test, and roll back trivially; curated guides
  stay pristine.
  - Cons: new injection + formatting plumbing; risk of learnings
  drifting/conflicting with the guide if never reconciled.

  Option C — both: machine writes to a learnings layer, vetted tips get promoted
   into curated guides
  - Pros: exactly matches your "auto-stage, human-promote" choice — the
  learnings layer is the staging area, promotion is the human gate. Safe
  auto-write, clean permanent guides, full provenance + rollback, bounded token
  cost.
  - Cons: most plumbing (a learnings store + a promotion tool).

  I recommend C. It's the only option that's coherent with the writeback model
  you already picked: auto-staging needs somewhere safe to land that isn't the
  curated guide, and "human-promote" needs a curated destination to promote
  into. A and B each give you only one half of that. C also directly serves your
   latency concern — the learnings injection is capped and confidence-gated, so
  distilled notes can't silently bloat prompts.

  One honest flag on scale: with the LLM piloting every decision, a single game
  is many sequential LLM calls, so "thousands of games" is throughput-bound.
  Mitigations I'll bake into the plan: pilot with a fast/small model
  (STRATEGIST_MODEL_NAME already supports a separate model), lean hard on the
  existing response cache, run games in parallel across multiple sidecar
  workers, and do reflection per batch (aggregate over N games → one distilled
  lesson set) rather than per game — fewer LLM calls and far more signal than
  reflecting on a single noisy game.

● Strong instincts — the poisoning risk is real and worth designing around
  explicitly. Here's my thinking.

  The anti-poisoning frame: store the unobstructed line, not "ignore 
  interaction"

  The danger isn't the aggressive prompt itself — it's an unconditional lesson
  ("commit everything, don't hedge") leaking into production reasoning against a
   control deck. You defend against that with two gates and a reframe:

  1. Quarantine the prompt as a mode, not the production prompt. Add a
  goldfish/solve pilot mode that swaps in a system prompt like "the opponent has
   zero interaction; find the fastest possible kill, do not hedge for 
  counterspells or removal." This prompt only exists inside the simulator —
  production /recognize never loads it. No code path connects them.
  2. Tag every learned lesson with the context it was learned in. A goldfish
  lesson is stored as conditional, never absolute: "Fastest unobstructed kill: 
  T2 reducer → T3 ritual chain → win (goldfish-derived, assumes no 
  interaction)." It is literally a different sentence from "ignore
  counterspells."
  3. The reframe that makes it safe AND useful: goldfishing computes your deck's
   theoretical fastest clock / nut line assuming no resistance. That's an upper 
  bound, not a directive. Production reasoning anchors to that line and then
  degrades from it based on observed interaction. So the lesson feeds the "plan
  turns ahead" system as a baseline plan — "here's the kill if unobstructed;
  deviate only as much as the opponent's shown interaction demands." Against 60
  Islands you execute it fully; against a control deck showing untapped blue
  mana, the strategist already hedges. You're not telling it to ignore
  counterspells — you're giving it the optimum it's hedging away from.
  4. Injection gating + human promotion (already in the plan): goldfish lessons
  inject only when state matches their context (opponent non-interactive / no
  interaction shown), and a human rewrites them conditionally before they ever
  reach a curated guide.

  Generic harness vs. goldfishing-only

  I'd build the general self-play harness and treat goldfishing as configuration
   #1, not a separate system. Goldfishing is just the special case where the
  opponent profile = "passive, no interaction." It's barely more code (the
  runner takes an opponent-deck arg and a pilot-mode arg), and the reflection →
  learnings → promotion pipeline is identical regardless of opponent. Building
  goldfish-only would force a rewrite the moment you want a "vs control" or "vs
  aggro gauntlet" run.

  And goldfishing specifically earns its place: no-interaction = zero opponent
  variance = the cleanest possible signal for "what is my optimal line?" For
  combo/aggro that's exactly where the nut sequencing falls out, which is the
  highest-value input to the turns-ahead planner. Matchup gauntlets (vs real
  decks) are noisier and produce conditional lessons — valuable, but second.

● Good idea — but let me be precise about why, because the "double the lessons
  in the same time" framing has one catch worth naming.

  The catch: your scarce resource is LLM inference, and a two-sidecar game makes
   ~2x the LLM calls per game (both seats query the sidecar) — so it's not a
  free 2x on wall-clock when inference is the bottleneck, which it is. In a
  goldfish game the passive opponent burns ~zero LLM calls; making that seat a
  second sidecar player doubles the inference cost of that game.

  Why it's still worth doing: the win isn't inference savings, it's (a) two
  labeled trajectories per engine game — you amortize all the Forge
  setup/sim/orchestration overhead across two lessons — and (b) genuinely
  interactive lessons from a real opponent making real plays, which is strictly
  higher quality than learning vs a scripted gauntlet AI. If you were going to
  run interactive gauntlets anyway, making both seats sidecar-piloted is the
  obvious upgrade.

  Two important constraints:
  - Mirror matches serve the matchup/conditional-lesson goal, not the
  optimal-line goal. Goldfishing's value is zero opponent variance = clean
  "fastest unobstructed line" signal; a mirror match reintroduces variance. So
  keep the passive 60-Islands config for optimal-line learning — don't spend LLM
   on a seat that should do nothing.
  - Both seats must use the normal/production prompt, never solve. Two players
  both told "ignore interaction, race" would generate garbage lessons. The
  quarantined aggressive prompt stays goldfish-only.

  So: it becomes a third config (sidecar-vs-sidecar mirror, normal mode), and it
   can subsume the scripted control gauntlet since sidecar-vs-sidecar is better
  data. Reflection just needs to attribute each game to both seats separately —
  which the per-(archetype, opponent-context) batching already handles.

     Self-Play Improvement Harness for the LLM Sidecar (Goldfishing + Gauntlet)

     Context

     The sidecar (forge-llm-sidecar/) advises Forge's AI via piloting guides
     (app/knowledge/piloting/<format>/<slug>.json, injected by
     game_advisor._format_guide, game_advisor.py:116). Today those guides are
     hand-authored. We want a closed loop: have the LLM sidecar pilot a deck
     through automated games, measure performance (turns-to-win), run a
     post-game
     reflection that distills what worked, auto-stage lessons to a learnings
     layer, and let a human promote vetted lessons into the curated guides.

     This is built as a general, parameterized self-play harness with three run
     configs:
     - (A) Goldfish — opponent is passive (60-Islands punching bag), pilot uses
     the quarantined solve prompt. Zero opponent variance → cleanest signal for
     a
     deck's optimal / fastest unobstructed line; highest-value input to the
     "plan turns ahead" system, especially combo/aggro. Only one seat burns LLM.
     - (B) Mirror / sidecar-vs-sidecar — BOTH seats are LLM-sidecar players
     using
     the normal/production prompt. Yields TWO labeled trajectories per engine
     game
     (amortizes Forge setup/sim overhead) and produces genuinely interactive
     matchup lessons. Costs ~2x LLM calls per game (both seats query the
     sidecar),
     so it's not a free throughput 2x — but it's strictly better data per engine
     game and supersedes a scripted control gauntlet.
     - (Optional) scripted gauntlet — cheaper (one seat LLM) but lower-quality
     opponent than (B); skip in favor of (B) unless inference budget forces it.

     Goldfish (A) feeds the optimal-line goal; Mirror (B) feeds the
     matchup/conditional-lesson goal. They are complementary, not substitutes —
     never run mirror games in solve mode (two recklessly-racing players
     generate
     garbage lessons).

     Anti-poisoning principle (core design constraint)

     We must NOT teach the production AI to "ignore counterspells / never
     hedge."
     Defenses:
     1. Quarantined pilot prompt. A solve/goldfish pilot mode swaps in an
     aggressive system prompt ("opponent has zero interaction; find the fastest
     kill, don't hedge"). This prompt exists ONLY in the simulator; production
     /recognize never loads it — no shared code path.
     2. Lessons are stored conditionally, tagged with their learned context. A
     goldfish lesson reads "Fastest unobstructed kill:  (goldfish-derived,
     assumes no interaction)" — never "ignore counterspells."
     3. Reframe: goldfishing yields the deck's theoretical fastest clock (an
     upper bound). Production anchors to it and degrades from it based on
     observed interaction. The turns-ahead planner gets the optimum to hedge
     away from, not a directive to play recklessly.
     4. Injection gating + human promotion: goldfish lessons inject only when
     game state matches their context (opponent non-interactive / no interaction
     observed); a human rewrites them conditionally before they reach a guide.

     Token / latency budget (drives all sizing)

     Latency = prefill (input tokens, ~linear) + decode (output tokens, the
     usual
     bottleneck). Sidecar emits small JSON, so input/prefill dominates — exactly
     what guidance size controls. Local llama.cpp prefill rough: ~0.7s (2k) /
     ~3.3s
     (10k) / ~33s (100k) on a decent GPU; 10-50x worse on CPU. 100k is a
     non-starter
     per call. KV prefix caching makes static guidance ~free after first call if
     it's a stable prefix.
     - Per-call dynamic content (game state): < ~2-4k tokens.
     - Curated guide injection: ~1,500-2,500 tokens (only acceptable as cached
     prefix).
     - Learnings injection: hard cap ~500 tokens, confidence/sample-gated.
     - Reflection output per lesson: ≤ ~500 tokens, distilled — never raw logs.

     Decisions (confirmed with user)

     - Pilot: LLM sidecar drives plays (max sidecar influence), to test the real
     system, plus post-game reflection.
     - Target: separate learnings layer that auto-stages; vetted lessons
     human-promoted into curated guides.
     - Writeback: auto-stage to learnings; human-promote into guides.
     - Scope (v1): parameterized harness shipping TWO configs — goldfish (A,
     passive 60 Islands, solve mode) AND mirror (B, sidecar-vs-sidecar, normal
     mode). Mirror replaces the scripted gauntlet for interactive lessons.
     - Consumer: optimal line feeds BOTH the turns-ahead planner (as a baseline)
     AND conditional learnings injection.

  llama.cpp slots for two sidecars

  Two sidecars pointing at the same llama-server share one model, so:
  - Yes, start llama-server with --parallel 2 (a.k.a. -np 2) so it has 2 slots
  and can process both sidecars' requests concurrently. With the default
  --parallel 1, the two sidecars' calls serialize (queue behind each other) —
  still correct, just no concurrency, so a gauntlet turn pair takes ~2× as long.
  - Slot assignment is automatic — llama.cpp picks a free slot per request; you
  do not set a slot number per sidecar. (There's an optional id_slot request
  param to pin a request to a slot for KV-cache continuity, but the OpenAI
  /v1/chat/completions path doesn't expose it and you don't need it.)
  - Context is split across slots: with --parallel 2, each slot gets n_ctx/2.
  Your model reports ~156k max, so each slot still has ~78k — far more than our
  small prompts need. If you'd set a small -c, bump it so n_ctx/2 stays
  comfortably above a few k.
  - Keep continuous batching on (--cont-batching, default on in recent builds)
  so the two streams interleave efficiently. Throughput for 2 concurrent
  requests on a 27B won't be a full 2× (they share GPU compute), but it's much
  better than serialized.

  Net: just relaunch llama-server with --parallel 2; everything else is
  automatic.
