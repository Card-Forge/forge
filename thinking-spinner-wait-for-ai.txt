     Plan: "AI is Thinking" indicator + clarification on guidance prompt size

     Context

     The user had two related concerns:

     1. Prompt size from guidance docs — worry that the whole
     guidance/modern.json was being injected into the LLM prompt and that we
     should extract only the matchup-relevant slice once recognition confidence
     is high.
     2. No visible feedback during sidecar waits — when the AI pauses to call
     the sidecar, the user has no signal that the AI is actually thinking vs.
     hung.

     Finding on (1) — no code change needed

     Investigation showed the forge-llm-sidecar/guidance/<format>.json files
     (the ~300KB hand-authored docs) are dead code today — never loaded at
     runtime. The live pipeline already does the equivalent of what the user
     proposed:

     - app/knowledge/loader.py:34 loads only the lean
     app/knowledge/archetypes/<format>.json (name, colors, signature_cards,
     strategy, tells).
     - app/nodes/game_advisor.py::_trim_to_meta_coverage (lines 466–500) shrinks
      the archetype list to the smallest set covering 90% of metagame share
     (floor 15, ceiling 40).
     - Per-archetype piloting guides
     (app/knowledge/piloting/<format>/<slug>.json) are loaded only after the
     AI's own archetype is identified deterministically (piloting.py:219-250),
     and _format_guide (game_advisor.py:111) compresses them to ~240 tokens.
     - The /recognize prompt totals ~1,700–2,500 tokens, of which the archetype 
     list is only 150–500 tokens; the dominant cost (~1,500–1,800 tokens) is the
      ranking-rules instruction block.

     So a confidence-gated "switch to one archetype's guidance" optimization
     would save ~200–400 tokens at most, and the user agreed this isn't worth
     touching code for. Faster inference (smaller model or
     enable_thinking=false) on the service side is the better lever — already
     configurable via llm_disable_thinking in llm_client.py:32-50. No sidecar 
     code changes in scope.

     Scope of this plan

     Only the "AI is thinking…" UI indicator on the Forge game screen, plus a
     one-line ack that the guidance extraction work was dropped.

     Approach

     Hook into the central sidecar-wait method (AiController.waitForSidecar) to
     publish a "thinking" status message into the existing prompt panel (CPrompt
      / VPrompt), then clear it when the wait returns. All UI updates marshalled
      to the EDT.

     This avoids any new endpoint, WebSocket, or sidecar protocol change. The
     spinner appears whenever the AI thread is blocked on
     SidecarInfluence.awaitLatest(), regardless of which decision point
     triggered it (mulligan, declare attackers, declare blockers, choose spell
     ability).

     Critical files

     - forge-ai/src/main/java/forge/ai/AiController.java (around line 146,
     waitForSidecar) — the single chokepoint for sidecar waits. Add notify-start
      / notify-end calls around the existing SidecarInfluence.awaitLatest(...)
     invocation.
     - forge-ai/src/main/java/forge/ai/llm/SidecarInfluence.java — already
     exposes the future state; reuse awaitLatest semantics.
     - forge-gui-desktop/src/main/java/forge/screens/match/CMatchUI.java (around
      line 1064, showPromptMessageNoCancel) — the existing reusable channel for
     transient status text in the prompt area.
     - forge-gui-desktop/src/main/java/forge/screens/match/controllers/CPrompt.j
     ava:50 and views/VPrompt.java:54 — the prompt panel the message renders
     into; no UI surgery required.
     - forge/util/FThreads (invokeInEdtNowOrLater) — used throughout CMatchUI;
     must wrap any UI mutation since the caller is on the AI worker thread.

     Implementation steps

     1. Add a small, package-private interface SidecarStatusListener (or reuse
     an existing event) in forge-ai/src/main/java/forge/ai/llm/ with
     onThinkingStart() / onThinkingEnd(). Keep it dependency-free so forge-ai
     doesn't have to import desktop UI.
     2. In AiController.waitForSidecar, before calling awaitLatest, fire
     onThinkingStart; in a finally block, fire onThinkingEnd. Use a static
     registry (SidecarInfluence already has module-static state — same pattern).
     3. In forge-gui-desktop, register a listener from CMatchUI at match start
     that calls showPromptMessageNoCancel("AI is thinking…") on start and
     restores the prior prompt text on end. Marshal both via
     FThreads.invokeInEdtNowOrLater.
     4. Guard against rapid start/end churn by only showing the message once the
      wait exceeds ~250 ms (use a Timer scheduled at start, cancelled on end).
     This avoids flicker for cache-hit /recognize calls (<100 ms).
     5. Restore the previously-shown prompt message on onThinkingEnd so we don't
      clobber game-flow prompts (getPromptMessage() save/restore).

     Out of scope (recorded for future)

     - Phase-level status (e.g. "recognizing opponent", "choosing line") — would
      need a new sidecar status endpoint/SSE, deferred.
     - Confidence-gated archetype-only mode (was suggested at threshold 0.65) —
     dropped because the savings (~200–400 tokens) don't justify the code churn;
      revisit if prompts grow.
     - Activating the guidance/<format>.json long-form docs — separate
     initiative, not blocking anything today.

     Verification

     1. Build forge-ai + forge-gui-desktop: mvn -pl forge-ai,forge-gui-desktop 
     -am install -DskipTests.
     2. Launch desktop: mvn -pl forge-gui-desktop exec:java (or via the run
     skill for this project).
     3. Start a match against an AI opponent in Modern with the sidecar running.
      Confirm:
       - At the start of an AI mulligan and on each AI turn, the prompt panel
     briefly shows "AI is thinking…" while the sidecar call is in flight.
       - The message disappears as soon as the AI makes its decision; the prior
     prompt text returns.
       - With sidecar stopped (or SIDECAR_BASE_URL pointing at a dead host), no
     spinner appears (DeckRecognitionClient short-circuits on health check) and
     the game still plays.
       - Fast cache hits (re-recognize on same state) do NOT flash the spinner
     because of the 250 ms grace.
     4. Confirm no UI freezes by running an entire AI turn and watching the
     prompt update via the EDT (no off-EDT Swing access warnings in the logs).

  Summary of changes:

  - New: forge-ai/src/main/java/forge/ai/llm/SidecarStatusBus.java —
  process-wide listener bus with Listener interface and fail-soft fire methods.
  - AiController.waitForSidecar (forge-ai) — wraps

  SidecarStatusBus.fireThinkingStart() and fireThinkingEnd().
  - CMatchUI (forge-gui-desktop) — registers a listener in the constructor and
  implements onSidecarThinkingStart / onSidecarThinkingEnd. The indicator only
  appears after a 250 ms grace (avoids flicker on cache hits), saves and
  restores lastPromptMessage, and is reentrant via an AtomicInteger depth
  counter. Listener is unregistered in finishGame when the match is over.

  Net behavior: when the AI blocks on a sidecar /recognize call longer than 250
  ms (mulligan, declare attackers/blockers, choose spell ability), the prompt
  panel shows "AI is thinking…" and reverts when the wait completes. No sidecar
  protocol or prompt-size changes — the guidance-extraction work was dropped
  because investigation showed guidance/<format>.json is dead code today, and
  the live pipeline already trims to ~150–500 tokens of archetype data (with
  ~1,500 of fixed instructions dominating the prompt).
